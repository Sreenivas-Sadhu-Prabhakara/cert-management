Specification & Field Manual · v1.0

One contract.
Four implementations.
Zero ambient trust.

A custody service for the riskiest object in any TLS deployment — the private key — from generation through CSR, cryptographically verified certificate-chain ingestion, activation, compromise and crypto-shredded deletion. Implemented identically in Java, Go, Node.js and Rust over one PostgreSQL schema, with an append-only audit trail enforced by the database itself.

§ 01 — The Premise

The contract is the trust anchor, not the code.

Most services ask you to trust an implementation. This one doesn't. Four programs, written independently in four languages with four dependency trees, implement a single normative specification against a single database. Any client may speak to any backend and observe identical status codes, identical JSON, identical cryptography — byte-for-byte fingerprints, interchangeable AES-GCM ciphertext, one shared JWT realm.

No single runtime or supply chain is load-bearing. Replace any of them overnight; the verified behaviour is what persists. That is Zero Trust applied to your own stack.

§ 02 — The Lifecycle

A key is not usable at birth. Trust is earned by verification.

The state machine makes the secure path the only path. A freshly generated key has no proof that any authority vouches for it — so activation is only reachable through a cryptographically verified certificate chain. There is no flag to skip it, no admin override, no side door.

● COMPROMISED

Reachable from any live state. Cannot be deleted, cannot be reactivated. A compromised key's record is incident evidence — containment must never enable cover-up.

○ DELETED

Soft delete: the ciphertext is destroyed — crypto-shredding — but the row, fingerprint, certificate metadata and full audit history remain. The secret dies; the accountability does not.

RULE — every transition is an atomic compare-and-set in PostgreSQL (UPDATE … WHERE status IN (…)); concurrent operators cannot race a key into an illegal state.

§ 03 — Chain Ingestion

Nothing private ever crosses the boundary inward. And even public material must prove itself.

This service deliberately has no private-key import. The only thing it ingests is the CA-signed certificate chain for a key it generated itself — and before that chain is stored, three proofs are demanded, in order, first failure wins:

PUBLIC-KEY BINDING422 · KEY_MISMATCH

The leaf certificate's SubjectPublicKeyInfo must be byte-identical to the stored public key. The right certificate for the wrong key — by mistake or by substitution — is refused at the door.

CHAIN INTEGRITY422 · CHAIN_BROKEN

Every certificate must be signed by its successor: signature verified, issuer/subject distinguished names linked, and a trailing self-signed root must verify its own signature. "Probably fine" is not a verdict.

VALIDITY WINDOWS422 · CERT_NOT_VALID

Expired or not-yet-valid certificates anywhere in the chain are rejected now — not discovered in production later. Every rejection is itself recorded in the audit trail.

§ 04 — Why It Must Be This Way

Every decision maps to a Zero Trust principle.

Zero Trust (NIST SP 800-207) abandons the trusted interior: authenticate every request, verify every artifact, record every action, assume breach. The architecture is the argument:

Design decisionPrinciple it serves
Keys born inside, never importedPrivate keys are generated within the service boundary; the only ingestion is public material, cryptographically verified. Never trust, always verify — the provenance of every private key is known with certainty. Nothing is taken on faith.
JWT on every requestHS256, issuer and expiry checked; 15-minute tokens; constant-time credential comparison. No ambient trust. Being on the network grants nothing; identity is re-proven per request and expires quickly.
AES-256-GCM at rest, AAD = record UUIDEach ciphertext is cryptographically bound to its own database row. Assume breach. A stolen dump yields ciphertext; even partial write access can't swap one key's ciphertext into another's record undetected.
Dedicated, audited /private endpointList and detail responses never carry private material; every private read writes PRIVATE_KEY_ACCESSED. Least privilege, full visibility. The dangerous operation is deliberate, separated, and leaves a trace every single time.
Append-only audit, enforced by triggerAudit rows are written in the same transaction as the state change; UPDATE/DELETE on the audit table raises an exception. Evidence over memory. Even the services themselves cannot rewrite history; a state change cannot commit without its audit event.
Soft delete = crypto-shreddingKey material destroyed, record retained. Inventory is a security control. "We don't know what keys we've had" is how outages and orphaned trust happen.
Four interchangeable implementationsOne OpenAPI contract, one schema, one conformance suite. Trust the contract, not the code. Components are replaceable and continuously verified — never pets.

the lifecycle, in four callscurl
# 1 · authenticate — every subsequent request carries this token
TOKEN=$(curl -s $BASE/api/v1/auth/token -d '{"clientId":"…","clientSecret":"…"}' | jq -r .accessToken)

# 2 · generate a key pair → CREATED  (the only response carrying privateKeyPem)
curl -s $BASE/api/v1/keys -H "Authorization: Bearer $TOKEN" \
     -d '{"name":"web-frontend-2026","algorithm":"EC_P256"}'

# 3 · CSR for your CA → send it out-of-band, get a signed chain back
curl -s $BASE/api/v1/keys/$ID/csr -H "Authorization: Bearer $TOKEN" \
     -d '{"subject":{"commonName":"www.example.test"},"sans":["www.example.test"]}'

# 4 · upload the chain → verified → READY_TO_PUBLISH → activate → ACTIVE
curl -s $BASE/api/v1/keys/$ID/certificate -H "Authorization: Bearer $TOKEN" \
     -d "{\"certificateChainPem\": $(jq -Rs . < chain.pem)}"
curl -s -X POST $BASE/api/v1/keys/$ID/activate -H "Authorization: Bearer $TOKEN"

§ 06 — The Contract

Twelve endpoints, identical on every port.

GET/healthLiveness + database connectivity. The only other unauthenticated route is the token endpoint.
POST/api/v1/auth/tokenClient credentials → short-lived HS256 bearer token.
POST/api/v1/keysGenerate RSA 2048/3072/4096 or EC P-256/P-384 → CREATED. Sole response containing privateKeyPem.
GET/api/v1/keys?status=…Inventory, newest first — deleted rows included, by design.
GET/api/v1/keys/{id}Full detail: public key, certificate, chain. Never private material.
GET/api/v1/keys/{id}/privateDecrypted private key. EVERY ACCESS AUDITED
POST/api/v1/keys/{id}/csrPKCS#10 signed by the stored key — subject DN + DNS SANs.
POST/api/v1/keys/{id}/certificateSigned-chain ingestion: binding → integrity → validity → READY_TO_PUBLISH.
POST/api/v1/keys/{id}/activateOnly from READY_TO_PUBLISH. There is no other road to ACTIVE.
POST/api/v1/keys/{id}/compromiseTerminal. Blocks deletion, blocks private-key reads, freezes the record as evidence.
DEL/api/v1/keys/{id}Soft delete — crypto-shreds the ciphertext, keeps the evidence row. Refused for compromised keys.
GET/api/v1/keys/{id}/auditThe append-only timeline, oldest first, tagged with the implementation that handled each event.
KEY_MISMATCHLeaf certificate's public key ≠ stored public key.
CHAIN_BROKENSignature or DN linkage failure inside the chain.
CERT_NOT_VALIDA certificate outside its validity window.
INVALID_STATEOperation illegal for the key's current status.
INVALID_PEMChain that does not parse as X.509 PEM.
UNAUTHORIZEDMissing or invalid credentials or token.
Trust nothing. Verify everything. Record it all.
